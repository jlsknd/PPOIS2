## 2. Постановка задачи

Разработать оконное приложение для учета товаров на складе.

### 2.1 Структура записи

| Поле | Тип данных | Описание |
|------|------------|----------|
| Название товара | Строка | Наименование товара |
| Название производителя | Строка | Наименование компании-производителя |
| УНП производителя | Строка | Учетный номер плательщика (может содержать ведущие нули) |
| Количество на складе | Целое число | Остаток товара на складе |
| Адрес склада | Строка | Физический адрес хранения |

### 2.2 Условия поиска и удаления

Пользователь может выбрать один из трех критериев:

1. **Критерий 1:** по названию товара или количеству на складе
2. **Критерий 2:** по названию производителя или УНП производителя
3. **Критерий 3:** по адресу склада

### 2.3 Основные требования к приложению

- Построение по шаблону Model-View-Controller
- Хранение записей в базе данных SQLite
- Возможность сохранять и загружать данные в/из XML файла
- DOM парсер для записи XML
- SAX парсер для чтения XML
- Постраничный вывод записей (пагинация)
- Отображение в виде таблицы и в виде дерева
- Поиск записей с выбором критерия в отдельном диалоговом окне
- Удаление записей с выбором критерия в отдельном диалоговом окне
- Сообщение о количестве удаленных записей
- Главное окно с меню и панелью инструментов
- Дочерние диалоговые окна для всех операций

## 3. Архитектура приложения

### 3.1 Паттерн Model-View-Controller (MVC)

Приложение реализовано с использованием архитектурного паттерна MVC, который разделяет программу на три компонента:

**Model (Модель)**
- Отвечает за хранение и управление данными
- Содержит бизнес-логику
- Работает с базой данных и XML файлами
- Не зависит от интерфейса

**View (Представление)**
- Отвечает за отображение данных пользователю
- Обрабатывает ввод пользователя
- Содержит элементы графического интерфейса
- Не содержит бизнес-логики

**Controller (Контроллер)**
- Обрабатывает действия пользователя
- Передает команды от View к Model
- Обновляет View после изменений в Model
- Связывает Model и View

### 3.2 Пассивная модель

В приложении используется пассивная модель, где View сама запрашивает данные у Model для обновления. Это упрощает архитектуру и делает код более предсказуемым.

## 5. Описание компонентов

### 5.1 model.py - Класс Product

```python
class Product:
    def __init__(self, name="", manufacturer_name="", manufacturer_unp="", 
                 quantity=0, address=""):
        self.id = None
        self.name = name
        self.manufacturer_name = manufacturer_name
        self.manufacturer_unp = manufacturer_unp
        self.quantity = quantity
        self.address = address
```

Класс описывает структуру одной записи о товаре. Все поля имеют соответствующие типы данных:
- `name`, `manufacturer_name`, `manufacturer_unp`, `address` - строковые
- `quantity` - целочисленный
- `id` - автоматически генерируется базой данных

## 6. Реализация функциональных требований

### 6.1 Хранение в базе данных SQLite

Таблица `products` создается при первом запуске приложения:

```sql
CREATE TABLE IF NOT EXISTS products (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    manufacturer_name TEXT NOT NULL,
    manufacturer_unp TEXT NOT NULL,
    quantity INTEGER NOT NULL,
    address TEXT NOT NULL,
    created_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP
)
```

Все операции с данными (добавление, поиск, удаление) выполняются через SQL запросы.

### 6.2 Работа с XML

**Сохранение (DOM парсер):**
- Создается корневой элемент `<inventory>`
- Для каждого товара создается элемент `<product>`
- Для каждого поля создается дочерний элемент
- XML записывается в файл с форматированием

**Загрузка (SAX парсер):**
- XML файл читается последовательно
- При обнаружении тега `<product>` создается объект Product
- При обнаружении полей заполняются соответствующие атрибуты
- При закрытии тега `<product>` товар добавляется в базу данных


### 6.3 Постраничный вывод (пагинация)

Реализован в главном окне и в диалоге поиска:

- Хранение `current_page` (текущая страница) и `page_size` (количество записей на странице)
- Вычисление `start_idx = current_page * page_size`
- Вычисление `end_idx = min(start_idx + page_size, total_products)`
- Отображение только записей с `start_idx` по `end_idx`
- Кнопки навигации: первая, предыдущая, следующая, последняя
- Возможность изменения количества записей на странице (5, 10, 15, 20, 50)
- Отображение номера текущей страницы и общего количества страниц
- Отображение общего количества записей

### 6.4 Отображение в виде таблицы и дерева

**Таблица (QTableWidget):**
- 5 колонок: Название товара, Производитель, УНП, Количество, Адрес склада
- Чередование цветов строк для удобства чтения
- Автоматическое растягивание колонок по содержимому

**Дерево (QTreeWidget):**
- Каждый товар - корневой элемент с текстом "Товар #ID"
- Поля товара - дочерние элементы (Название товара, Производитель, УНП, Количество, Адрес)
- Две колонки: "Поле" и "Значение"
- Корневые элементы развернуты по умолчанию

### 6.5 Поиск и удаление с выбором критерия

**Выбор критерия через радиокнопки:**
- Вариант 1: По названию товара или количеству
- Вариант 2: По названию производителя или УНП
- Вариант 3: По адресу склада

**Динамическое изменение полей ввода:**
- При выборе критерия 1: активны поля "Название товара" и "Количество"
- При выборе критерия 2: активны поля "Название производителя" и "УНП"
- При выборе критерия 3: активно поле "Адрес склада"
- Остальные поля отключаются

**Поиск:**
- Формируется SQL запрос с учетом выбранного критерия
- Используется оператор LIKE для частичного совпадения (для текстовых полей)
- Используется оператор = для точного совпадения (для УНП и количества)
- Результаты отображаются в том же диалоге с пагинацией

**Удаление:**
- Аналогично поиску формируются критерии
- Выполняется DELETE запрос
- Показывается сообщение о количестве удаленных записей
- Если записи не найдены - соответствующее сообщение
